--------------------------- Introduction ---------------------------
Programming Paradigm: A fundamental Style or Philosophy of writing and structuring and reasoning about code.
   │
   ├── Imperative (HOW to do things)
   │   ├── Procedural
   │   └── Object-Oriented
   │
   └── Declarative (WHAT result you want)
       ├── Functional
       ├── Logic
       └── Dataflow / Reactive

Imperative Paradigm: (How to do things)
0. Basic Imperative Paradigm (The Origin): You describe step-by-step instructions that change program state.
   The program executes statements sequentially.
Problems:
  1. State Explosion: Large programs → too many variables → unpredictable state.
  2. Spaghetti Code: Control jumps everywhere (especially in early BASIC/C).
  3. Hard Maintenance: Changing one part breaks others.


1. Procedural Programming Paradigm: Functions/Procedures, Modularity, Scope Control
   Instead of writing everything in one file, we organize code into reusable procedures.
Problems:
  Data and behavior are separate: Data structures are separate from functions that operate on them.
    This causes:
    Tight coupling
    Difficult scaling
    Large codebases become chaotic


2. OOP Paradigm:
Core Idea
Model real-world entities as objects.
Bundle Inside an object:
   Data (state)
   Behavior (methods)
Classifications (Properties/Behaviors[Actions]) => Objects (Values of Properties/Do Actions)
Problems:
  1. Inheritance Hell: Deep class hierarchies become fragile.
  2. Mutable State Everywhere: Hard to reason about bugs.
  3. Side Effects: Objects modifying shared state → race conditions.
  4. Tight Coupling: This caused industry to rethink architecture.


3. Functional Programming Paradigm:
Core Philosophy: Computation = evaluation of mathematical functions.
Core Concepts
  Pure functions: A function where the same input always produces the same output, and it has no side effects
  Immutability
  First-class functions
  Higher-order functions: Functions that operate on other functions.
Problems:
  1. Steep Learning Curve: Math-heavy concepts.
  2. Performance Overhead: Immutability can increase memory usage.
  3. Hard Debugging in Lazy Systems: This led to hybrid approaches.


4. Hybrid Approach Paradigm

Declarative Paradigm: (What result you want)
0. Basic Declarative Paradigm
Core Idea: You describe WHAT you want, not HOW to compute it.
Example: SQL => SELECT * FROM users WHERE age > 18;

1. Logic Programming: Define Facts, and rules => System derive answers
Example: You define relationships => The engine infers.
  parent(ali, lamiaa).
  parent(omar, salma).
  grandparent(X, Y) :- parent(X, Z), parent(Z, Y).

Event-Driven Programming: Execution is driven by events.
Example in JavaScript:
  button.addEventListener("click", () => {
    console.log("Clicked");
  });

Reactive Programming: When data changes → UI updates.
Used in: Angular(Signals) - React - Vue.js - RxJS

Data-Oriented Programming: Focus on Memory layout - CPU cache efficiency - Performance optimization
Used in: Games Engines - High-performance systems

Multi-Paradigms:
  Python => Procedural + OOP + Functional
  JavaScript => Imperative + Functional + Event-Driven
  Scala => OOP + Functional
  Rust => Systems + Functional

2. Virtual DOM + Reconciliation:
DOM = Document Object Model
Traditional DOM update:
  DOM change → Browser re-render entire UI
  HTML
  ↓
  DOM Tree
  Updating DOM is expensive.


React:
  Virtual DOM = JavaScript representation of the realDOM in memory.
  Virtual DOM → diff → minimal update

The DOM (Document Object Model) is slow to manipulate directly.
React uses a Virtual DOM, which is:
	A lightweight, in-memory copy of the real DOM.
	Represented as a JavaScript object tree.
When the state changes:
	React creates a new Virtual DOM tree.
	It compares (diffs) the new tree with the previous one.
	It updates only the changed elements in the real DOM (this process is called Reconciliation).

React uses a powerful engine called the Fiber Architecture to manage this efficiently — it breaks updates into small units for better performance and smoother rendering.

Execution pipeline:
  User action
  ↓
  State changes
  ↓
  Virtual DOM Update
  ↓
  Diff Algorithm (Reconciliation) => React compares old vs new
  ↓
  Minimal Real DOM update (Only necessary changes)
  ↓
  UI Change

Algorithm name:
  Reconciliation

Stack Reconciler => O(n^3)
React Fiber Reconciler => O(n)


Example Concept:
Real DOM:
  <ul>
    <li>Apple</li>
    <li>Orange</li>
  </ul>

User adds item.

New UI:
  <ul>
    <li>Apple</li>
    <li>Orange</li>
    <li>Mango</li>
  </ul>

React compares both versions and updates only the new element.
Instead of rebuilding the entire list.

Advantages:
  Faster rendering
  Efficient updates
  Explain Reconciliation algorithm.
  Predictable UI behavior

JSX Rules:
Rule 1 — Return only one Component (Single/One Parent Element)
  ❌
  <h1>Hello</h1>
  <p>Text</p>

  ✅
  <div>
    <h1>Hello</h1>
    <p>Text</p>
  </div>

Recommended: Use <React.Fragment> </React.Fragment> => <> </>

Rule 2 — Convert any Similar keywords from HTML to camelCase Style
  HTML ❌  => JSX ✅
  tabindex  => tabIndex
  onclick   => onClick
  onchange  => onChange
  maxlength => maxLength
  readonly  => readOnly

Rule 3 — Use className instead of class, htmlFor instead of for

Rule 4 — Use JavaScript Expressions => Any JavaScript syntax inside HTML part after return written inside {}
  {2 + 2}

Rule 5 — Self-Closing Tags
<img> => <img/>


Rule 6 — Inline Styles
  HTML: ❌
  <div style="background-color: blue; font-size: 16px;"></div>

  JSX: ✅
  <div style={{ backgroundColor: 'blue', fontSize: '16px' }}></div>

Rule 7 — {/* This is a comment */}

Rule 8 — Boolean Attributes
HTML: <input type="checkbox" checked>
JSX: <input type="checkbox" checked /> (or checked={true})

# How to Use Vite v8 (JavaScript + React Compiler), React v19.2, Tailwindcss v4.2, and Shadcn UI:

1. Terminal: npm create vite@latest

2. Terminal: npm install tailwindcss @tailwindcss/vite

3. Replace everything in src/index.css with the following:
@import "tailwindcss";

4. Create jsconfig.json inside root (not in ./src)
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@/*": ["./src/*"]
    }
  }
}


5. Update vite.config.js

import { defineConfig } from 'vite';
import react, { reactCompilerPreset } from '@vitejs/plugin-react';
import babel from '@rolldown/plugin-babel';
import tailwindcss from '@tailwindcss/vite';

// https://vite.dev/config/
export default defineConfig({
  plugins: [react(), babel({ presets: [reactCompilerPreset()] }), tailwindcss()],
  resolve: {
    alias: {
      '@': new URL('./src', import.meta.url).pathname,
    },
  },
});

import { createBrowserRouter } from "react-router";

const router = createBrowserRouter([
	{
		path: '/login',
		element: <LoginFormPage />,
	},
	{
		path: '/signup',
		element: <SignupFormPage />,
	},
	{
		path: '/dashboard',
		element: <DashboardPage />,
	},
]);


createRoot(document.getElementById('root')).render(
		<StrictMode>
			<RouterProvider router={router} />
		</StrictMode>
);
